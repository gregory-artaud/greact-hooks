# Publish 0.1.4 Recovery Design

## Goal

Publish `greact-hooks@0.1.4` with the `useDramaticPause` feature that was present in the failed `v0.1.3` tag, while preventing future pnpm setup version conflicts.

## Root Cause

The `v0.1.3` workflow configured pnpm twice: `pnpm/action-setup` requested the major version `10`, while `package.json` declared the exact version `pnpm@10.28.2`. Version conflict validation in `pnpm/action-setup@v4` stopped the job before dependency installation and publication.

Re-running that workflow cannot fix the failure because GitHub Actions reuses the workflow stored in the tagged commit.

## Design

Apply the feature commit referenced by `v0.1.3` to `main` without moving or deleting the failed tag. Remove the `version` input from `pnpm/action-setup@v4`, leaving `package.json#packageManager` as the single pnpm version source. Change the package version from `0.1.3` to `0.1.4` after applying the feature commit.

The existing `v0.1.3` tag remains an immutable record of the failed release attempt. A new annotated `v0.1.4` tag will trigger the corrected publish workflow.

## Verification

Before tagging:

- Install dependencies with the frozen lockfile.
- Run all tests, including the `useDramaticPause` tests.
- Build the library and verify generated declarations.
- Run `pnpm pack --dry-run` and confirm the package reports version `0.1.4` and includes the expected distribution files.
- Confirm the workflow has no explicit pnpm version input.

After pushing `main` and `v0.1.4`, monitor the triggered GitHub Actions run to completion. Confirm that the publish job succeeds and that the npm registry reports `greact-hooks@0.1.4`.

## Failure Handling

If pre-tag verification fails, do not create or push the tag. If the remote workflow fails after the tag is pushed, preserve the tag and report the failing step instead of moving it or publishing manually.

## Scope

This recovery includes the already-reviewed `useDramaticPause` commit, the workflow correction, the version bump, and the release tag. It does not alter the failed `v0.1.3` tag or introduce unrelated feature changes.
